# Memory, Bandwidth, Latency Quiz

## 객관식 1
어떤 팀이 긴 문맥 서비스에서 모델 폭을 줄여 FLOPs를 낮췄지만 `TPOT` 개선은 3%뿐이었다. 이때 가장 먼저 세워야 할 가설은 무엇인가?

1. 현재 지배 항이 compute보다 `bytes moved` 쪽이라서, 모델 축소 효과가 제한됐다.
2. `prefill`이 항상 `decode`보다 느리므로 첫 토큰 문제만 보면 된다.
3. utilization이 높으면 자동으로 compute-bound다.
4. 모델 폭 축소가 실패했다면 attention 수식 자체를 바꿔야 한다.

## 객관식 2
Roofline 관점에서 arithmetic intensity가 낮은 workload를 가장 잘 설명한 것은 무엇인가?

1. 메모리에서 옮기는 바이트에 비해 계산이 적어서 bandwidth ceiling이 더 낮은 상한이 되기 쉽다.
2. 계산 유닛 사용률이 높으므로 반드시 compute-bound다.
3. `prefill`에만 적용되고 `decode`에는 적용되지 않는다.
4. 모델 파라미터 수만 줄이면 자동으로 arithmetic intensity가 높아진다.

## 객관식 3
다음 중 `[S3]`와 `[S4]`에서 직접 확인 가능한 사실만으로 묶인 문장은 무엇인가?

1. `[S3]`는 최신 serving 문서 기준선이고, `[S4]`는 `AWS Neuron release notes 2.26.0`이다.
2. `[S3]`와 `[S4]`는 모든 추론 서비스에서 bandwidth ceiling을 항상 제일 먼저 보라고 직접 지시한다.
3. `[S4]`는 긴 문맥 decode 병목이 전부 KV cache라고 단정한다.
4. `[S3]`는 compute 최적화를 중단하라고 직접 결론 내린다.

## 짧은 서술형 1
`prefill`과 `decode`를 하나의 병목으로 묶어 진단하면 왜 잘못된 처방이 나오는지 2~3문장으로 설명하라. 답안에는 `TTFT`와 `TPOT`을 반드시 포함하라.

## 짧은 서술형 2
이 챕터의 Part 6처럼, 아래 두 항목을 2~3문장으로 구분해 써라.

- `[S3][S4]`에서 직접 확인 가능한 사실 1개
- 그 사실 위에 놓는 운영 해석 1개

답안에는 `사실`, `해석`이라는 단어를 반드시 포함하라.

## 심화 설명형 1
두 서비스가 있다.

- 서비스 A: `메일 제목 자동 라우터`, 문맥이 짧고 답변도 짧음
- 서비스 B: `32K 계약 조항 비교`, 문맥이 길고 답변 중 근거 문맥을 오래 유지해야 함

두 서비스 모두 사용자는 `느리다`고 말한다. 그런데 같은 최적화 순서를 적용하면 안 되는 이유를 설명하라. 답안에는 `prefill`, `decode`, `연산량 최적화`, `데이터 이동 최적화`, `bandwidth ceiling`을 포함하라.

## 심화 설명형 2
어떤 팀이 다음처럼 보고했다.

`모델 축소와 커널 튜닝을 했는데도 개선폭이 작고, 문맥 길이가 길어질수록 뒤쪽 토큰이 더 느려진다.`

이 상황에서 왜 `compute 최적화를 더 밀기 전에 bytes moved를 다시 세야 하는지` 설명하라. 답안에는 `arithmetic intensity`, `Roofline`, `문맥 길이 민감도`, `TPOT`을 포함하라.

## 정답 및 해설
### 객관식 정답
- 객관식 1: 1번
- 객관식 2: 1번
- 객관식 3: 1번

### 객관식 해설
- 객관식 1: 모델 축소는 주로 FLOPs를 줄이지만, 긴 문맥 서비스가 이미 `bytes moved / bandwidth` 쪽 상한에 가까우면 개선폭은 작을 수 있다. 이때는 최적화 실패보다 지배 항 오판 가능성을 먼저 봐야 한다.
- 객관식 2: arithmetic intensity는 `FLOPs / Bytes Moved`다. 값이 낮을수록 같은 시간에 계산보다 메모리 이동이 더 강한 제약이 되므로 bandwidth ceiling을 먼저 의심해야 한다.
- 객관식 3: 이 챕터에서 직접 확인 가능한 최신성 사실은 문서의 성격과 기록된 날짜 수준까지다. `무조건 bandwidth부터 보라` 같은 결론은 본문이 그 위에 올린 해석이지, `[S3][S4]`의 직접 문장으로 취급하면 안 된다.

### 서술형 예시 답안
- 짧은 서술형 1 예시: `prefill`은 긴 입력 전체를 처리하며 `TTFT`를 크게 흔들고, `decode`는 새 토큰을 반복 생성하며 `TPOT`을 크게 흔든다. 둘을 하나로 묶으면 긴 입력 중복을 줄여야 할 상황과 긴 문맥 read를 줄여야 할 상황을 구분하지 못해 처방이 빗나간다.
- 짧은 서술형 2 예시: `사실`로는 `[S3]`가 최신 serving 문서 기준선이고 `[S4]`가 `AWS Neuron release notes 2.26.0`이라는 점을 들 수 있다. `해석`으로는 이 문서들이 보여 주는 최신성 때문에 성능 진단에서도 정적 FLOPs 표만 보지 말고 실행 경로와 `bytes moved`를 함께 점검해야 한다고 말할 수 있다.

### 심화 설명형 채점 포인트
- 심화 설명형 1: 서비스 A는 짧은 문맥이라 `decode`의 긴 read보다 `prefill` 고정비나 요청 경로가 더 중요할 수 있고, 따라서 `연산량 최적화`가 먼저일 수 있다는 점이 들어가야 한다. 서비스 B는 긴 문맥 때문에 `decode`에서 `bandwidth ceiling`과 `데이터 이동 최적화`가 더 우선일 수 있다는 구분이 필요하다.
- 심화 설명형 2: 답안은 `문맥 길이 민감도`가 커질수록 `TPOT` 악화가 `bytes moved` 증가와 연결될 수 있음을 설명해야 한다. 또한 `Roofline`과 `arithmetic intensity`를 사용해, compute 튜닝 개선폭이 작을 때는 지배 항이 compute가 아닐 수 있으므로 bytes moved 재측정이 다음 단계라는 판단을 보여 줘야 한다.
